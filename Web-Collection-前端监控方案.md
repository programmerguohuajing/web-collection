# Web Collection 自研前端监控平台

> 项目名称：Web Collection 前端监控平台  
> 版本日期：2026-07-15

## 一句话结论

自研前端监控平台的核心价值，是把“用户在页面上发生了什么、页面为什么慢、错误为什么发生、如何复现问题”沉淀为公司自己的可查询、可回放、可分析的数据资产。

当前 Web Collection 已从浏览器监控扩展为跨端采集底座：统一覆盖 Web、微信/支付宝/抖音等小程序，以及 uni-app、Taro、React Native 应用。

第三方平台适合快速启动，但在埋点口径、用户维度、私有化数据、成本可控、错误回放、业务定制看板等方面会逐步受限。

## 一、为什么要做前端监控

前端是用户真实体验的最后一公里。服务端接口正常，不代表用户页面正常；页面能打开，也不代表用户操作顺畅。

| 监控方向 | 解决的问题 | 对业务的价值 |
| --- | --- | --- |
| 行为埋点 | 用户访问了哪些页面、点击了哪些按钮、在哪一步流失 | 支撑运营分析、产品决策、转化漏斗优化 |
| 性能监控 | 页面首屏慢、接口慢、资源慢、交互卡顿 | 定位体验瓶颈，降低等待和流失 |
| 错误监控 | JS 报错、Promise 异常、资源加载失败、接口异常 | 缩短故障发现和修复时间 |
| 会话回放 | 只看错误栈无法复现用户操作路径 | 还原用户真实操作，减少沟通成本 |
| SourceMap 反解 | 线上压缩代码无法直接定位源码 | 把线上错误定位到源码文件、行、列 |

对于小程序和 App，平台继续解决页面访问、业务行为、请求耗时、运行错误和用户定位问题。由于非 Web 运行时没有浏览器 DOM，小程序与原生 App 不启用 rrweb；浏览器端仍保留完整的 rrweb 会话回放。

## 二、自研带来的核心收益

### 1. 从被动反馈变成主动发现

没有监控时，问题通常来自用户投诉、客服转述或测试偶现。自研监控后，可以主动发现：

- 哪个版本错误量突然升高
- 哪个页面 LCP、FCP、TTFB 变差
- 哪些接口请求耗时异常
- 哪些静态资源加载失败
- 哪个用户在哪个页面触发了错误

### 2. 从猜测原因变成证据定位

单条错误日志通常只能告诉我们“报错了”。完整监控链路可以提供：

- 错误信息、错误栈、文件行列号
- SourceMap 反解后的源码位置
- 用户行为栈：报错前点击、路由、接口、曝光等行为
- rrweb 会话回放：还原用户真实操作过程
- 用户维度：userId、用户名、手机号脱敏展示、应用、版本、URL

### 3. 从工具能力变成公司数据资产

第三方平台的数据口径由第三方产品决定，自研平台的数据口径由业务决定。例如：

- 订单提交、支付点击、合同上传、商品发布等业务动作可以按公司字段沉淀
- 用户维度可以和内部系统打通
- 数据可以长期保留在公司数据库
- 后续可以接入内部 BI、告警、工单、发布系统

## 三、为什么不直接使用第三方平台

第三方平台并不是不能用，而是不适合作为长期完全依赖的核心监控底座。

| 维度 | 第三方平台 | Web Collection 自研 |
| --- | --- | --- |
| 上线速度 | 快，接 SDK 即可 | 前期需要建设 |
| 数据归属 | 数据进入第三方平台 | 数据保留在公司内部 |
| 成本控制 | 通常按数据量、PV、上报量、功能包计费 | 主要是服务器、数据库和存储成本 |
| 埋点模型 | 受平台字段和事件模型约束 | 可按业务模型自由设计 |
| 用户维度 | 通常支持，但字段、查询、脱敏受限 | 可按内部用户体系定制 |
| 错误定位 | 标准能力成熟 | 可按业务错误归因定制 |
| 会话回放 | 部分平台支持，通常与套餐相关 | 当前项目已接入 rrweb |
| 私有化 | 成本高或需要企业版 | 天然支持私有化部署 |
| 二次开发 | 受平台开放能力限制 | 完全可控 |
| 合规风险 | 需评估数据出域、用户隐私、供应商合规 | 可在内网和自有数据库内闭环 |

建议定位：第三方平台可作为竞品参考或短期兜底，自研平台作为公司长期可控的前端监控基础设施。

## 四、与腾讯及其他厂商对比

腾讯侧能力需要分开看：前端性能/错误更接近腾讯云前端性能监控 RUM；用户行为埋点和分析更接近增长分析类服务。它们不是一个完全等价于“自研前端监控平台”的单一产品。

### 价格与计量方式

| 厂商 / 产品 | 计量口径 | 公开价格或额度 | 对我们的影响 |
| --- | --- | --- | --- |
| 腾讯云 RUM | 按上报数据量计费，PV、API、静态资源、错误日志、自定义上报都计入 | 单个主账号每天共享 50 万条免费上报量；超出后按公开计费单价计费 | 高频埋点、接口测速、资源测速都会推高上报量；多个业务统一接入后成本随规模增长 |
| 阿里云 ARMS 前端监控 | PV 上报、API 上报、自定义上报、会话等分项计费 | 资源包按上报次数或会话量计费 | 页面复杂、API 多、埋点多时成本明显 |
| Datadog RUM | 按用户 Session 计费，Session Replay 独立计量 | 价格随区域、套餐、年付/月付变化 | 国际化能力强，但更适合已有 Datadog 体系的团队 |
| Sentry | 按错误、Trace Span、Replay、日志、Metric 等事件量计费 | 付费计划包含默认额度，超额后增加预算 | 错误监控成熟，但行为埋点和业务分析不是核心定位 |
| Web Collection | 自有事件、会话、错误、性能数据入公司数据库 | 不对外按事件付费 | 可按业务价值决定采样、保留周期、冷热分层和字段模型 |

### 成本敏感性示例

以按上报量计费的平台为例，假设每天 100 万 PV，每个页面平均产生：

- 2 次 API 性能上报
- 2 次资源性能上报
- 0.2 次自定义行为上报

则每日上报量约为：

```text
100 万 PV + 200 万 API + 200 万资源 + 20 万自定义 = 520 万条/日
```

按腾讯云 RUM 公开口径（每天 50 万条免费，超出部分 0.34 元/万条）估算：

```text
计费量：520 万 - 50 万 = 470 万条/日
日成本：470 × 0.34 = 159.8 元
月成本：159.8 × 30 = 4,794 元/月
```

该测算仅包含示例中的页面、API、资源和自定义事件，不包含更多业务埋点、错误日志及会话回放。价格依据 2026-07-15 查询到的腾讯云官方公开计费文档，正式采购应以当时官网报价为准。

如果再开启更多自定义行为、错误面包屑、用户路径、会话回放，第三方账单会继续随数据量增长。自研平台虽然也有存储和计算成本，但可以通过采样、批量上报、字段裁剪、回放抽样、数据保留周期来主动治理，不会被外部平台的事件单价锁定。

### 自定义能力对比

| 维度 | 腾讯等第三方平台 | Web Collection 自研 |
| --- | --- | --- |
| 事件字段 | 支持标准事件和自定义属性，但受产品模型约束 | 事件结构、用户字段、业务字段完全可控 |
| 查询条件 | 依赖平台提供的查询维度 | 支持时间、appId、release、URL/path、userId、用户名、手机号、关键字、状态等 |
| 数据处理 | 平台内置聚合和分析 | 可按公司业务规则清洗、脱敏、聚合、归档 |
| SDK 行为 | 以平台 SDK 能力为准 | Web 按 error、performance、behavior、replay、fetch、xhr、WebSocket、SSE 拆分；跨端复用统一事件协议 |
| 回放策略 | 取决于平台能力和套餐 | 支持 rrweb 录制和回放，可控制路由切换分段、最长录制时间 |
| 源码定位 | 通常支持 SourceMap 上传 | 支持 SourceMap 上传和错误反解 |
| 私有化部署 | 需看产品是否支持及报价 | 当前架构天然支持私有化部署 |

## 五、当前项目已具备的能力

### SDK 采集能力

| 模块 | 当前能力 |
| --- | --- |
| 行为埋点 | PV、点击、路由、停留、滚动、曝光、自定义 track |
| 性能监控 | FCP、LCP、CLS、TTFB、INP/FID、longtask、resource、接口耗时 |
| 错误监控 | JS 错误、Promise 异常、资源错误、接口错误 |

| 会话回放 | rrweb 录制，支持路由切换分段、最长录制时间、敏感区域屏蔽 |
| TypeScript | Web 与跨端入口均提供类型声明，支持 TypeScript 项目直接导入 |
| 跨端 SDK | 支持微信、支付宝、抖音、百度、QQ、快手、京东小程序，以及 uni-app、Taro、React Native |

| 构建产物 | Web 输出 ES Module / IIFE；跨端输出 ES Module / CommonJS，全部支持压缩混淆 |
| SDK 托管 | API 服务支持 Web 与跨端构建文件，以及 `/sdk/...` 统一托管路径 |

### 跨端接入架构

| 运行端 | 接入入口 | 自动或统一能力 | 运行时边界 |
| --- | --- | --- | --- |
| Web | `@web-collection/sdk` | DOM 行为、Web Vitals、资源、fetch/xhr、WebSocket/SSE、错误、rrweb | 支持完整浏览器会话回放 |
| 通用小程序 | `@web-collection/sdk/miniapp` | 自动识别 `wx/my/tt/swan/qq/ks/jd`，统一 request、存储、页面和全局错误 | 无 DOM，不启用 rrweb |
| uni-app | `@web-collection/sdk/uni-app` | 复用 `uni.request`、存储与页面生命周期 | 无 DOM 端不启用 rrweb |
| Taro | `@web-collection/sdk/taro` | 复用 Taro request、页面生命周期与错误事件 | 由目标小程序或 App 运行时决定 |
| React Native | `@web-collection/sdk/react-native` | fetch、AsyncStorage、路由上下文、错误与性能事件 | 无浏览器 DOM，不启用 rrweb |

所有运行端统一上报到 `/api/collect`，复用相同的 appId、release、用户字段、sessionId、deviceId 和 PostgreSQL 数据模型。业务后台无需为不同终端维护多套查询系统。

## 六、构建与部署

```bash
pnpm install
pnpm build
pnpm --filter @web-collection/api db:init
pnpm --filter @web-collection/api start
```

构建后会生成统一产物目录：

```text
dist/
  apps/api/
  apps/web/dist/
  packages/sdk/dist/
```

SDK 目录包含：

```text
web-collection-sdk.es.js
web-collection-sdk.iife.js
web-collection-sdk.platform.js
web-collection-sdk.platform.cjs
```

生产服务默认端口为 `8787`。访问 `http://127.0.0.1:8787/` 可打开控制台，Web 与跨端 SDK 均可通过 `/sdk/<文件名>` 获取。

## 七、建设收益

### 对研发

- 错误从“用户说有问题”变成“后台直接定位”
- SourceMap 反解减少线上压缩代码排查成本
- rrweb 回放减少复现沟通成本
- fetch/xhr、WebSocket/SSE、资源、性能指标自动采集，降低手工排查成本
- 一套事件协议覆盖 Web、小程序和 App，减少各技术栈重复建设

### 对业务

- 通过行为数据了解用户真实路径
- 可分析页面访问、按钮点击、转化步骤和流失位置
- 支持按用户维度查询，便于定位重点客户问题

### 对管理

- 形成统一监控入口，而不是每个项目各自零散排查
- 数据归公司所有，便于长期积累和复盘
- 成本可控，可按业务优先级决定采样率、保留周期和扩容节奏

## 八、后续建议

| 优先级 | 建议 | 价值 |
| --- | --- | --- |
| P0 | 增加告警规则和通知 | 错误量、性能劣化能主动触达 |
| P0 | 增加数据保留和清理策略 | 控制数据库增长 |
| P1 | 增加应用管理和版本管理 | 支撑多业务接入 |
| P1 | 增加 SourceMap 自动上传脚本 | 接入 CI/CD 后自动关联版本 |
| P1 | 选择一个 Web 业务和一个移动端业务试点 | 验证跨端统一事件与用户查询价值 |
| P2 | 增加采样策略配置后台 | 控制高流量业务成本 |
| P2 | 增加报表导出 | 支持周报、月报和复盘 |

## 汇报结论

1. 自研不是重复造轮子，而是把前端体验数据变成公司自己的数据资产。
2. 第三方适合快速启动，但长期会受成本、数据、字段模型和私有化限制。
3. 当前项目已经完成 Web、主流小程序、uni-app、Taro、React Native SDK，以及后端、看板、SourceMap、浏览器会话回放和生产托管的核心闭环。
4. 下一步建议选择 Web 与移动端核心业务试点，并补齐告警、数据保留和采样治理。
